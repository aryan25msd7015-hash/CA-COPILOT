Phase 1 — Infrastructure & database (weeks 1–2)

Build the multi-tenant data foundation. Every table gets an org_id. Get this architecture wrong and you'll rewrite it under production load.

Step 1
Provision RDS PostgreSQL 16 + extensions
Launch RDS PostgreSQL 16 in ap-south-1. Enable automated backups (7-day). Use db.t3.medium for dev, db.r6g.large for prod. Connect and immediately enable three extensions.
-- Run after first connection CREATE EXTENSION IF NOT EXISTS vector; -- for RAG embeddings CREATE EXTENSION IF NOT EXISTS "uuid-ossp"; -- for UUID PKs CREATE EXTENSION IF NOT EXISTS pg_trgm; -- for fuzzy matching
Step 2
Core multi-tenant schema
Every single table must include org_id as a foreign key. This is the most critical architectural decision in the entire project — it is what prevents firm A from ever seeing firm B's data.
CREATE TABLE organizations ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), name TEXT NOT NULL, plan TEXT DEFAULT 'starter', gstin TEXT, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE clients ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), org_id UUID REFERENCES organizations(id) ON DELETE CASCADE, name TEXT NOT NULL, gstin TEXT, whatsapp_number TEXT, whatsapp_consent_at TIMESTAMPTZ, email TEXT, health_score INT DEFAULT 100, industry TEXT, -- needed for benchmarking created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE users ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), org_id UUID REFERENCES organizations(id) ON DELETE CASCADE, email TEXT UNIQUE NOT NULL, role TEXT CHECK (role IN ('partner','manager','article')), created_at TIMESTAMPTZ DEFAULT NOW() );
Step 3
Documents & transactions tables
CREATE TABLE documents ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), org_id UUID REFERENCES organizations(id), client_id UUID REFERENCES clients(id), doc_type TEXT, -- invoice,gstr2b,purchase_register,notice,trial_balance,bank_statement s3_key TEXT NOT NULL, ocr_text TEXT, ocr_json JSONB, source TEXT DEFAULT 'upload', -- 'upload' or 'whatsapp' status TEXT DEFAULT 'pending', created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE transactions ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), org_id UUID REFERENCES organizations(id), client_id UUID REFERENCES clients(id), document_id UUID REFERENCES documents(id), invoice_no TEXT, vendor_gstin TEXT, vendor_name TEXT, amount NUMERIC(15,2), tax_amount NUMERIC(15,2), date DATE, match_status TEXT DEFAULT 'unmatched', match_confidence NUMERIC(5,2), anomaly_score NUMERIC(5,4), source TEXT DEFAULT 'upload', created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE legal_chunks ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), doc_type TEXT, -- income_tax_act, gst_act, circular, reply_template content TEXT, embedding vector(1536), created_at TIMESTAMPTZ DEFAULT NOW() );
Step 4
FastAPI + tenant middleware
The tenant middleware injects org_id into every request automatically. Every DB query must go through the scoped helper — never query a table without the org_id filter.

Phase 4 — GST & ITC reconciliation engine (weeks 5–8)

The anchor module — the one CAs will pay for on day one. A three-tier matching engine that handles real-world data messiness: vendor name typos, rounding errors, and date shifts.

Step 1
Data normalization — run before any matching
def normalize(df): df['invoice_no'] = (df['invoice_no'].astype(str).str.upper() .str.replace(r'[^A-Z0-9/\-]','',regex=True)) df['vendor_name'] = (df['vendor_name'].str.upper().str.strip() .str.replace(r'\b(PVT|LTD|PRIVATE|LIMITED|LLP)\b','',regex=True) .str.replace(r'\s+',' ',regex=True).str.strip()) df['date'] = pd.to_datetime(df['date'], dayfirst=True) df['amount'] = df['amount'].round(2) return df
Step 2
Three-tier matching waterfall
from rapidfuzz import fuzz def reconcile(purchase_df, gstr2b_df, config): amt_tol = config.get('amount_tolerance', 5) day_tol = config.get('date_tolerance', 3) fuzz_thr = config.get('fuzzy_threshold', 85) results, matched_p, matched_g = [], set(), set() # TIER 1 — Exact match: GSTIN + invoice_no + amount for _, p in purchase_df.iterrows(): match = gstr2b_df[ (gstr2b_df.vendor_gstin == p.vendor_gstin) & (gstr2b_df.invoice_no == p.invoice_no) & (gstr2b_df.amount == p.amount)] if len(match): results.append({**p, 'match_type':'exact', 'confidence':100.0, 'g_id':match.iloc[0].id}) matched_p.add(p.id); matched_g.add(match.iloc[0].id) # TIER 2 — Tolerance: amount ±₹5, date ±3 days rem_p = purchase_df[~purchase_df.id.isin(matched_p)] rem_g = gstr2b_df[~gstr2b_df.id.isin(matched_g)] for _, p in rem_p.iterrows(): match = rem_g[ (rem_g.vendor_gstin == p.vendor_gstin) & (abs(rem_g.amount - p.amount) <= amt_tol) & (abs((rem_g.date - p.date).dt.days) <= day_tol)] if len(match): results.append({**p,'match_type':'tolerance','confidence':90.0}) matched_p.add(p.id); matched_g.add(match.iloc[0].id) # TIER 3 — Fuzzy: Jaro-Winkler on vendor name rem_p2 = purchase_df[~purchase_df.id.isin(matched_p)] rem_g2 = gstr2b_df[~gstr2b_df.id.isin(matched_g)] for _, p in rem_p2.iterrows(): best_score, best_g = 0, None for _, g in rem_g2.iterrows(): s = fuzz.token_sort_ratio(p.vendor_name, g.vendor_name) if s > best_score: best_score, best_g = s, g if best_score >= fuzz_thr and abs(p.amount - best_g.amount) <= amt_tol: results.append({**p,'match_type':'fuzzy', 'confidence':float(best_score)}) unmatched = purchase_df[~purchase_df.id.isin(matched_p)] return pd.DataFrame(results), unmatched
Step 3
Isolation Forest anomaly detector
from sklearn.ensemble import IsolationForest def train_anomaly(client_id): txns = get_historical(client_id, months=6) stats = txns.groupby('vendor_gstin')['amount'].agg( ['mean','std','count']).fillna(0) model = IsolationForest(contamination=0.02, random_state=42) model.fit(stats[['mean','std','count']]) return model, stats def anomaly_score(model, stats, vendor_gstin, amount): s = stats[stats.index==vendor_gstin] if s.empty: return 0.75 # new vendor = moderate risk feat = s[['mean','std','count']].values raw = model.decision_function(feat)[0] return round(max(0, min(1, 1-(raw+0.5))), 4)

Phase 5 — Smart deadline engine (weeks 5–6)

Not just a calendar — a behavioral prediction system that sends personalized reminders based on each client's actual filing history. Chronic late filers get 10-day alerts, not 3-day.

Step 1
Compliance deadline database
CREATE TABLE compliance_deadlines ( id UUID PRIMARY KEY DEFAULT uuid_generate_v4(), org_id UUID REFERENCES organizations(id), client_id UUID REFERENCES clients(id), filing_type TEXT, -- GSTR1, GSTR3B, TDS_24Q, ROC_AOC4, ADV_TAX period TEXT, -- '2024-10' deadline DATE, status TEXT DEFAULT 'pending', -- pending, filed, missed filed_at TIMESTAMPTZ, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Pre-populate standard deadlines for all active clients -- GSTR-1: 11th of next month -- GSTR-3B: 20th of next month -- TDS Q1: 31 July, Q2: 31 Oct, Q3: 31 Jan, Q4: 31 May
Step 2
Behavioral lateness predictor
Look at a client's last 12 months of filing dates. Calculate their average days-late per filing type. Use this to personalize reminder lead time — don't treat every client the same.
def compute_lead_time(client_id: str, filing_type: str) -> int: history = db.query(ComplianceDeadline).filter( ComplianceDeadline.client_id == client_id, ComplianceDeadline.filing_type == filing_type, ComplianceDeadline.status.in_(['filed','missed']) ).order_by(ComplianceDeadline.deadline.desc()).limit(12).all() if len(history) < 3: return 7 # default: remind 7 days before avg_days_late = sum( max(0,(f.filed_at.date()-f.deadline).days if f.filed_at else 5) for f in history) / len(history) if avg_days_late >= 5: return 12 # chronic late filer if avg_days_late >= 2: return 9 # sometimes late return 5 # usually on time
Step 3
Client health contribution from deadlines
On-time filings add to the health score, missed deadlines subtract. This feeds into the Client Health Score module in Phase 7.
def deadline_health_component(client_id: str) -> float: """Returns 0.0 to 1.0 for the deadline component""" last_12 = get_last_12_filings(client_id) if not last_12: return 1.0 on_time = sum(1 for f in last_12 if f.status=='filed' and f.filed_at and f.filed_at.date() <= f.deadline) return round(on_time / len(last_12), 3)

Phase 6 — Tax notice analyzer & RAG drafter (weeks 9–12)

Upload a notice, get a legally grounded draft reply in minutes instead of hours. The model is constrained to only cite laws from your verified knowledge base — zero hallucination risk.

Step 1
Build & index the legal knowledge base
from langchain.text_splitter import RecursiveCharacterTextSplitter from openai import OpenAI LEGAL_DOCS = [ ("income_tax_act", "data/income_tax_act_1961.pdf"), ("gst_act", "data/cgst_act_2017.pdf"), ("circular", "data/gst_circulars_2024.pdf"), ("reply_template", "data/successful_replies_library.pdf"), ] splitter = RecursiveCharacterTextSplitter( chunk_size=400, chunk_overlap=50) def build_knowledge_base(): oai = OpenAI() for doc_type, path in LEGAL_DOCS: text = extract_pdf_text(path) chunks = splitter.split_text(text) for chunk in chunks: emb = oai.embeddings.create( model="text-embedding-3-small", input=chunk).data[0].embedding db.execute("""INSERT INTO legal_chunks (doc_type, content, embedding) VALUES (:dt,:c,:e)""", dict(dt=doc_type, c=chunk, e=emb))
Step 2
Notice parser — extract structured fields
NOTICE_RE = { 'notice_type': r'(Intimation|Scrutiny|Demand|Show Cause)', 'section': r'[Uu]nder [Ss]ection\s+(\d+[A-Za-z]*(?:/\d+[A-Za-z]*)*)', 'ay': r'A\.?Y\.?\s*(20\d{2}-\d{2,4})', 'demand_amt': r'(?:Rs|INR|₹)\.?\s*([\d,]+(?:\.\d{2})?)', 'due_date': r'(?:within|by|before)\s+(\d{1,2}[/-]\d{1,2}[/-]\d{2,4})', } def parse_notice(text: str) -> dict: return {k: (re.search(v,text) or type('',(),{'group':lambda s,i:None})()) .group(1) for k,v in NOTICE_RE.items()}

Phase 7 — Client health score (weeks 9–10)

A single 0–100 number per client combining GST compliance, TDS on-time rate, ITC mismatch gap, outstanding notices, and anomaly flags. CAs see instantly who needs urgent attention.

Step 1
Five-component scoring model
def compute_health_score(client_id: str) -> dict: # Component 1: GST on-time filing rate (last 12 months) gst_score = deadline_health_component(client_id) * 25 # Component 2: ITC reconciliation gap % recon = get_latest_reconciliation(client_id) itc_gap_pct = recon['unmatched_value'] / max(recon['total_value'],1) itc_score = max(0, 25 * (1 - min(itc_gap_pct * 5, 1))) # Component 3: Outstanding notices (each = -8 points, max -25) notices = count_open_notices(client_id) notice_score = max(0, 25 - (notices * 8)) # Component 4: Anomaly density (% of txns flagged) anomaly_rate = get_anomaly_rate(client_id) anomaly_score = max(0, 15 * (1 - min(anomaly_rate * 10, 1))) # Component 5: TDS compliance rate tds_score = get_tds_compliance_rate(client_id) * 10 total = round(gst_score+itc_score+notice_score+anomaly_score+tds_score) tier = "green" if total>=75 else "amber" if total>=50 else "red" return {"score": total, "tier": tier, "components": {"gst":gst_score,"itc":itc_score, "notices":notice_score,"anomaly":anomaly_score,"tds":tds_score}}

Phase 9 — Anomaly & fraud detection (weeks 13–16)

Scans ledger data for Benford's Law violations, round-number payments, weekend transactions, duplicate entries, and sudden vendor spikes. Generates a prioritized risk shortlist for the auditor.

Step 1
Benford's Law test
Genuine financial data follows Benford's Law: ~30% of amounts start with 1, ~18% with 2, etc. Fabricated data often doesn't. Flag clients where the distribution deviates significantly.
import numpy as np from scipy import stats BENFORD = [0.301,0.176,0.125,0.097,0.079,0.067,0.058,0.051,0.046] def benford_test(amounts: list) -> dict: first_digits = [int(str(abs(a))[0]) for a in amounts if a!=0] observed = [first_digits.count(d)/len(first_digits) for d in range(1,10)] chi2, p_val = stats.chisquare(observed, f_exp=BENFORD) return {"chi2": round(chi2,3), "p_value": round(p_val,4), "suspicious": p_val < 0.05, "deviation": [round(o-e,3) for o,e in zip(observed,BENFORD)]}

def flag_rule_anomalies(txns_df) -> pd.DataFrame: flags = [] # Round number payments (common in fabricated invoices) txns_df['flag_round'] = (txns_df['amount'] % 1000 == 0) & \ (txns_df['amount'] > 10000) # Weekend transactions txns_df['flag_weekend'] = pd.to_datetime( txns_df['date']).dt.dayofweek >= 5 # Duplicate invoice numbers txns_df['flag_duplicate'] = txns_df.duplicated( subset=['invoice_no','vendor_gstin'], keep=False) # Just-below threshold (₹49,999 to avoid ₹50,000 TDS limit) txns_df['flag_threshold'] = (txns_df['amount'] >= 45000) & \ (txns_df['amount'] < 50000) # Combined risk score txns_df['risk_flags'] = (txns_df['flag_round'].astype(int) + txns_df['flag_weekend'].astype(int) + txns_df['flag_duplicate'].astype(int)*3 + # weighted higher txns_df['flag_threshold'].astype(int)*2) return txns_df
