# Requirements Document

## Introduction

The CA Intelligence Platform is a multi-tenant SaaS product built for Chartered Accountant firms in India. It automates the most time-consuming workflows in a CA practice: document collection, GST reconciliation, compliance deadline tracking, tax notice drafting, anomaly detection, and client health monitoring. The platform integrates WhatsApp for client communication, Azure Document Intelligence for OCR, Claude for AI-assisted drafting, and PostgreSQL with pgvector for semantic search. All data is stored in AWS ap-south-1 (Mumbai) in compliance with India's Digital Personal Data Protection (DPDP) Act 2023. Every tenant (CA firm) is fully isolated at the database level via org_id on every table.

---

## Glossary

- **Organization (Org)**: A CA firm that subscribes to the platform. Every data record is scoped to an Org via `org_id`.
- **Client**: An individual or business entity whose accounts are managed by the CA firm.
- **Platform**: The CA Intelligence Platform system as a whole.
- **Tenant_Middleware**: The FastAPI middleware that extracts `org_id` and `role` from the JWT and injects them into every request.
- **Scoped_Query**: A database query helper that always applies the `org_id` filter before any data access.
- **Document**: A raw file (PDF, image, CSV) uploaded to or received by the Platform.
- **Transaction**: A structured financial record extracted from a Document.
- **OCR_Pipeline**: The asynchronous Celery task chain that processes Documents through Azure Document Intelligence.
- **PII_Masker**: The pre-storage component that redacts PAN numbers, Aadhaar numbers, and bank account numbers from extracted text.
- **Reconciliation_Engine**: The three-tier matching engine that compares purchase register transactions against GSTR-2B data.
- **Deadline_Engine**: The behavioral compliance calendar that generates and schedules personalized reminders.
- **WhatsApp_Pipeline**: The Meta Business API integration that sends outbound reminders and receives inbound documents from clients.
- **RAG_Drafter**: The Retrieval-Augmented Generation component that drafts replies to tax notices using a legal knowledge base stored in pgvector.
- **Health_Score_Engine**: The component that computes the 0–100 composite client health score.
- **Anomaly_Detector**: The component that applies Isolation Forest, Benford's Law, and rule-based heuristics to detect financial anomalies.
- **Invoice_Fraud_Scanner**: The component that validates GST invoices against live GSTIN data and mathematical tax rules.
- **NL_Query_Engine**: The natural language to SQL component that allows users to query their data in plain English.
- **Benchmarking_Engine**: The cross-client anonymized comparison component available on the premium tier.
- **Audit_Working_Papers**: ICAI-format structured audit documents generated from trial balance data.
- **GSTIN**: Goods and Services Tax Identification Number — a public business identifier, never masked.
- **GSTR-2B**: A static monthly ITC (Input Tax Credit) statement generated by the GST portal.
- **ITC**: Input Tax Credit — the credit a business can claim for GST paid on purchases.
- **PAN**: Permanent Account Number — a personal tax identifier, always masked before storage.
- **Aadhaar**: India's national biometric ID number — always masked before storage.
- **DPDP_Act**: Digital Personal Data Protection Act 2023 — India's primary data privacy legislation.
- **ICAI**: Institute of Chartered Accountants of India — the regulatory body that sets audit standards.
- **Celery_Beat**: The Celery periodic task scheduler used for daily compliance checks and reminder dispatch.
- **RBAC**: Role-Based Access Control. Three roles exist: `partner`, `manager`, `article`.

## Requirements

### Requirement 1: Multi-Tenant Data Isolation

**User Story:** As a CA firm partner, I want my firm's client data to be completely isolated from all other firms, so that there is no risk of data leakage between tenants.

#### Acceptance Criteria

1. THE Platform SHALL assign a unique UUID `org_id` to every Organization upon registration.
2. THE Platform SHALL include `org_id` as a non-nullable foreign key on every database table that stores business data.
3. WHEN a JWT-authenticated API request is received, THE Tenant_Middleware SHALL extract `org_id` and `role` from the token payload and attach them to the request context.
4. THE Scoped_Query SHALL filter every database query by the `org_id` from the request context before returning any results.
5. IF a database query is executed without an `org_id` filter on a multi-tenant table, THEN THE Platform SHALL raise an error and reject the query.
6. WHEN an Organization is deleted, THE Platform SHALL cascade-delete all records belonging to that Organization.

---

### Requirement 2: Authentication and Role-Based Access Control

**User Story:** As a CA firm partner, I want to control what each team member can see and do, so that article clerks cannot access sensitive partner-level functions.

#### Acceptance Criteria

1. THE Platform SHALL authenticate users via JWT tokens containing `org_id`, `user_id`, and `role`.
2. THE Platform SHALL enforce three roles: `partner`, `manager`, and `article`, where `partner` has the highest privilege.
3. WHEN a user with `article` role attempts to access a partner-only endpoint, THE Platform SHALL return HTTP 403 Forbidden.
4. WHEN a user with `manager` role attempts to access a partner-only endpoint, THE Platform SHALL return HTTP 403 Forbidden.
5. THE Platform SHALL allow `partner` role users to create, update, and delete other users within their Organization.
6. WHEN a JWT token is expired or invalid, THE Platform SHALL return HTTP 401 Unauthorized and reject the request.

---

### Requirement 3: Document Ingestion and S3 Upload

**User Story:** As a CA, I want to upload client documents (PDFs, images, CSVs) securely, so that they are stored safely without exposing my backend servers.

#### Acceptance Criteria

1. WHEN a user requests to upload a document, THE Platform SHALL generate an S3 pre-signed URL with a 300-second expiry scoped to the Organization's storage path.
2. THE Platform SHALL store documents in S3 using the path pattern `{org_id}/{client_id}/{uuid}.{ext}` to enforce tenant isolation at the storage layer.
3. THE Platform SHALL accept document types: `invoice`, `gstr2b`, `purchase_register`, `notice`, `trial_balance`, and `bank_statement`.
4. WHEN an S3 pre-signed URL is generated, THE Platform SHALL create a Document record in the database with status `pending` before returning the URL.
5. THE Platform SHALL support documents received via WhatsApp as an alternative ingestion source, recorded with `source = 'whatsapp'`.
6. WHEN a document is uploaded via S3, THE Platform SHALL enqueue an OCR task on the `ocr` Celery queue without blocking the upload response.

### Requirement 4: OCR Processing and PII Masking

**User Story:** As a CA, I want uploaded documents automatically processed and sensitive personal data redacted, so that I can query structured data without violating client privacy.

#### Acceptance Criteria

1. WHEN a Document with status `pending` is enqueued, THE OCR_Pipeline SHALL process it using Azure Document Intelligence with the `prebuilt-invoice` model.
2. THE OCR_Pipeline SHALL extract both raw text and structured key-value pairs and tables from each Document.
3. BEFORE storing any extracted text to the database, THE PII_Masker SHALL replace all PAN numbers matching the pattern `[A-Z]{5}[0-9]{4}[A-Z]` with a deterministic hash token.
4. BEFORE storing any extracted text to the database, THE PII_Masker SHALL replace all Aadhaar numbers matching the pattern `\d{4}\s?\d{4}\s?\d{4}` with a deterministic hash token.
5. THE PII_Masker SHALL NOT mask GSTIN values, as they are public business identifiers.
6. BEFORE passing any document content to an LLM API, THE PII_Masker SHALL be applied to the text.
7. WHEN OCR processing completes successfully, THE OCR_Pipeline SHALL update the Document status to `ocr_complete`.
8. WHEN OCR processing fails, THE OCR_Pipeline SHALL update the Document status to `ocr_failed` and log the error.
9. THE OCR_Pipeline SHALL run asynchronously in a Celery worker and SHALL NOT be executed synchronously within a FastAPI request handler.

### Requirement 5: Tally CSV Normalizer

**User Story:** As a CA, I want Tally-exported CSV files from both Tally Prime and Tally ERP 9 to be automatically normalized, so that I do not need to manually clean column headers or date formats.

#### Acceptance Criteria

1. THE Platform SHALL map Tally Prime column headers (`Voucher No`, `Party Name`, `GSTIN/UIN of Party`, `Amount`) to canonical field names (`invoice_no`, `vendor_name`, `vendor_gstin`, `amount`).
2. THE Platform SHALL map Tally ERP 9 column headers (`Vch No.`, `Particulars`, `Debit`) to the same canonical field names.
3. WHEN parsing date columns, THE Platform SHALL parse dates in `dayfirst=True` format (DD/MM/YYYY) as used in Indian accounting.
4. THE Platform SHALL remove commas from amount strings before converting to numeric and SHALL drop rows where the amount cannot be parsed.
5. THE Platform SHALL normalize vendor names by converting to uppercase, stripping leading/trailing spaces, and removing the suffixes `PVT`, `LTD`, `PRIVATE`, `LIMITED`, and `LLP`.

### Requirement 6: GST and ITC Reconciliation Engine

**User Story:** As a CA, I want to automatically reconcile my client's purchase register against GSTR-2B data, so that I can identify mismatches and ITC claim discrepancies without manual comparison.

#### Acceptance Criteria

1. WHEN a reconciliation is run, THE Reconciliation_Engine SHALL first attempt Tier 1 exact matching by comparing `vendor_gstin`, `invoice_no`, and `amount` fields exactly.
2. FOR all purchase records not matched in Tier 1, THE Reconciliation_Engine SHALL attempt Tier 2 tolerance matching where `vendor_gstin` matches exactly, amount difference is within a configurable tolerance (default ±₹5), and date difference is within a configurable tolerance (default ±3 days).
3. FOR all purchase records not matched in Tier 1 or Tier 2, THE Reconciliation_Engine SHALL attempt Tier 3 fuzzy matching using token sort ratio on vendor names, requiring a score above a configurable threshold (default 85) and amount within tolerance.
4. THE Reconciliation_Engine SHALL assign a `match_type` of `exact`, `tolerance`, or `fuzzy` and a confidence score to each matched transaction.
5. THE Platform SHALL store per-client reconciliation configuration including `amount_tolerance`, `date_tolerance`, and `fuzzy_threshold` in a `reconciliation_config` table.
6. WHEN a CA updates reconciliation configuration for a client, THE Platform SHALL apply the new thresholds to all subsequent reconciliation runs for that client.
7. THE Reconciliation_Engine SHALL produce an unmatched transactions list containing all purchase records with no match in GSTR-2B.
8. THE Platform SHALL allow export of reconciliation results including matched and unmatched transactions as an Excel file.
9. WHEN reconciliation is triggered, THE Platform SHALL run it as an asynchronous Celery task on the `heavy` queue.

### Requirement 7: Anomaly Detection within Reconciliation

**User Story:** As a CA, I want to see an anomaly risk score on every transaction, so that I can quickly identify suspicious or unusual entries for further investigation.

#### Acceptance Criteria

1. THE Anomaly_Detector SHALL train an Isolation Forest model per client using the last 6 months of historical transaction data, grouped by `vendor_gstin` with features: mean amount, standard deviation, and transaction count.
2. THE Anomaly_Detector SHALL assign an anomaly score between 0.0 and 1.0 to each transaction, where higher scores indicate greater anomaly risk.
3. WHEN a transaction involves a vendor GSTIN with no prior history, THE Anomaly_Detector SHALL assign a default anomaly score of 0.75 (moderate risk).
4. THE Platform SHALL store the anomaly score on the `transactions` table as `anomaly_score`.
5. THE Platform SHALL display the anomaly score alongside every transaction row in the reconciliation view.

---

### Requirement 8: Smart Deadline Engine

**User Story:** As a CA, I want a compliance calendar that sends personalized reminders to clients based on their filing behavior, so that chronic late filers receive earlier warnings automatically.

#### Acceptance Criteria

1. THE Platform SHALL maintain a compliance deadline database covering GST filings (GSTR-1, GSTR-3B), TDS returns (24Q, 26Q), advance tax dates, and ROC filing deadlines.
2. THE Deadline_Engine SHALL compute a `days_before_alert` value per client per filing type based on historical late-filing behavior.
3. WHEN a client has a history of filing late, THE Deadline_Engine SHALL set `days_before_alert` to a minimum of 10 days for that filing type.
4. WHEN a client has a consistent on-time filing history, THE Deadline_Engine SHALL set `days_before_alert` to a minimum of 3 days for that filing type.
5. THE Deadline_Engine SHALL run as a daily scheduled task via Celery Beat at 09:00 IST.
6. WHEN the current date is within `days_before_alert` of a compliance deadline and the required document has not been received, THE Deadline_Engine SHALL enqueue a WhatsApp reminder via the WhatsApp_Pipeline.
7. WHEN a required document is received for a filing, THE Deadline_Engine SHALL mark that document requirement as fulfilled and suppress further reminders for that filing cycle.

### Requirement 9: WhatsApp Client Communication Pipeline

**User Story:** As a CA, I want to send automated document collection reminders to clients on WhatsApp and receive their files directly, so that I do not need to manually follow up via phone or email.

#### Acceptance Criteria

1. BEFORE sending any WhatsApp message to a client, THE WhatsApp_Pipeline SHALL verify that the client record contains a non-null `whatsapp_consent_at` timestamp.
2. WHEN a client has not consented, THE WhatsApp_Pipeline SHALL NOT send any outbound message to that client.
3. THE Platform SHALL provide a consent URL endpoint that, when accessed by a client, records their `whatsapp_consent_at` timestamp in the database.
4. THE WhatsApp_Pipeline SHALL send all outbound scheduled messages using Meta Business API pre-approved message templates only.
5. THE Platform SHALL maintain three approved message templates: `data_request_reminder`, `data_received_ack`, and `filing_deadline_urgent`.
6. THE WhatsApp_Pipeline SHALL cap outbound reminder messages at 3 per client per filing cycle.
7. THE WhatsApp_Pipeline SHALL NOT send more than one reminder per client per day.
8. WHEN a client sends a STOP reply, THE Platform SHALL immediately update `whatsapp_consent_at` to null and cease all future outbound messages to that client.
9. WHEN a WhatsApp webhook POST is received, THE Platform SHALL return HTTP 200 within 200 milliseconds and process the message asynchronously via BackgroundTasks.
10. WHEN a client sends a document via WhatsApp, THE Platform SHALL download the media, upload it to S3, create a Document record with `source = 'whatsapp'`, and enqueue the OCR task.
11. WHEN a document is received and processed via WhatsApp, THE WhatsApp_Pipeline SHALL send an acknowledgment message to the client with the count of extracted transactions.
12. WHEN an unregistered phone number sends a message, THE Platform SHALL reply with a registration guidance message and SHALL NOT process the content.

### Requirement 10: Tax Notice RAG Drafter

**User Story:** As a CA, I want to draft replies to tax notices with AI assistance grounded in actual GST and Income Tax law, so that I can respond faster and with accurate legal citations.

#### Acceptance Criteria

1. THE Platform SHALL maintain a legal knowledge base (`legal_chunks` table) containing chunked content from the Income Tax Act, GST Act, CBDT circulars, and reply templates with 1536-dimensional vector embeddings.
2. WHEN a tax notice document is submitted for drafting, THE RAG_Drafter SHALL retrieve the top-k most semantically similar legal chunks from the knowledge base using cosine similarity on pgvector.
3. THE RAG_Drafter SHALL pass the retrieved legal chunks and the notice text to Claude (claude-sonnet-4) with a system prompt that limits the response to information supported by the retrieved context.
4. WHEN Claude generation is complete, THE RAG_Drafter SHALL validate that every legal citation in the generated reply exists verbatim in the retrieved legal chunks.
5. IF a generated citation cannot be verified against the retrieved chunks, THEN THE RAG_Drafter SHALL remove that citation and flag it as unverified in the output.
6. THE Platform SHALL display the final draft alongside the source legal chunks that were used, so the CA can verify the citations.
7. THE RAG_Drafter SHALL run as an asynchronous Celery task on the `llm` queue.
8. IF the primary LLM (Claude) is unavailable, THEN THE Platform SHALL fall back to OpenAI and log the fallback event.

### Requirement 11: Client Health Score

**User Story:** As a CA partner, I want a single 0–100 score for each client summarizing their overall compliance health, so that I can prioritize attention across my entire client portfolio.

#### Acceptance Criteria

1. THE Health_Score_Engine SHALL compute a composite score on a scale of 0 to 100 for each client.
2. THE Health_Score_Engine SHALL compute the score from five weighted components: GST compliance rate, ITC gap percentage, open tax notices count, anomaly density, and TDS compliance rate.
3. WHEN any component input changes (e.g., a new notice is uploaded, reconciliation completes), THE Health_Score_Engine SHALL recompute the affected client's health score.
4. THE Platform SHALL store the current health score on the `clients.health_score` column.
5. THE Platform SHALL display the health score on the client list view with a visual indicator distinguishing high-risk clients (score below 50), medium-risk clients (score 50–74), and healthy clients (score 75–100).
6. THE Platform SHALL maintain a historical record of health score changes per client with timestamps.

### Requirement 12: AI Audit Working Papers

**User Story:** As a CA conducting a statutory audit, I want to generate ICAI-format working papers from a client's trial balance, so that I can reduce the time spent on routine audit documentation.

#### Acceptance Criteria

1. THE Platform SHALL accept trial balance files in CSV or Excel format and parse them into structured debit/credit ledger entries.
2. THE Platform SHALL compute standard financial ratios from the trial balance including current ratio, debt-to-equity ratio, gross profit margin, and net profit margin.
3. WHEN trial balance data is available, THE Platform SHALL use Claude to generate ICAI-format audit observations covering materiality assessment, variance analysis, and key risk areas.
4. THE Platform SHALL export the generated audit working papers as a DOCX file formatted according to ICAI audit documentation standards.
5. WHEN a trial balance file cannot be parsed, THE Platform SHALL return a descriptive error message identifying the parsing failure.
6. THE Platform SHALL NOT include any client PII in the DOCX export beyond what was present in the original trial balance.

### Requirement 13: Advanced Anomaly and Fraud Detection

**User Story:** As a CA, I want the system to automatically flag suspicious patterns in my client's financial data, so that I can detect potential fraud or errors before they cause compliance issues.

#### Acceptance Criteria

1. THE Anomaly_Detector SHALL apply Benford's Law analysis to transaction amounts, flagging datasets where the first-digit distribution deviates significantly from the expected logarithmic distribution.
2. THE Anomaly_Detector SHALL flag transactions with amounts that are round numbers (multiples of ₹1,000 above ₹10,000) as requiring review.
3. THE Anomaly_Detector SHALL flag transactions dated on weekends (Saturday or Sunday) as requiring review.
4. THE Anomaly_Detector SHALL detect duplicate transactions by identifying records with the same vendor GSTIN, invoice number, and amount within a 30-day window.
5. THE Anomaly_Detector SHALL detect threshold gaming by identifying clusters of transactions from the same vendor that are each just below a reportable threshold (e.g., multiple entries just below ₹50,000).
6. THE Anomaly_Detector SHALL detect vendor spikes by flagging vendors whose transaction volume increases by more than 300% month-over-month.
7. THE Platform SHALL display all anomaly flags on a dedicated anomaly dashboard per client, sortable by flag type and risk score.
8. THE Anomaly_Detector SHALL run as an asynchronous Celery task triggered after each reconciliation run completes.

### Requirement 14: Invoice Fraud Scanner

**User Story:** As a CA, I want each uploaded invoice automatically validated against live GSTIN data and tax arithmetic rules, so that I can catch fraudulent or erroneous invoices before filing.

#### Acceptance Criteria

1. WHEN an invoice document is processed, THE Invoice_Fraud_Scanner SHALL validate the vendor GSTIN against the live GST portal API to confirm the GSTIN is active.
2. THE Invoice_Fraud_Scanner SHALL verify that the tax amount on the invoice equals the taxable amount multiplied by the declared tax rate (CGST + SGST or IGST) within a tolerance of ₹1.
3. THE Invoice_Fraud_Scanner SHALL detect duplicate invoices by computing a fingerprint from `vendor_gstin + invoice_no + amount` and checking for prior occurrences within the client's transaction history.
4. WHEN a GSTIN validation, tax math check, or duplicate check fails, THE Invoice_Fraud_Scanner SHALL set a `fraud_flag` on the transaction record with the specific failure reason.
5. THE Platform SHALL display all fraud-flagged invoices in a dedicated review queue for the CA to action.
6. THE Invoice_Fraud_Scanner SHALL run asynchronously in a Celery task after OCR processing completes.

### Requirement 15: Natural Language Query Engine

**User Story:** As a CA, I want to ask questions about my client data in plain English and receive accurate results, so that I do not need to write SQL or depend on pre-built reports.

#### Acceptance Criteria

1. WHEN a user submits a natural language query, THE NL_Query_Engine SHALL translate the query into a SQL SELECT statement targeting only the authenticated Organization's data.
2. THE NL_Query_Engine SHALL inject the `org_id` filter into every generated SQL query before execution, regardless of the query content.
3. THE NL_Query_Engine SHALL execute only SELECT statements and SHALL reject any generated query containing INSERT, UPDATE, DELETE, DROP, or other data-modification operations.
4. WHEN a generated SQL query fails to execute, THE NL_Query_Engine SHALL return a human-readable error message and SHALL NOT expose raw SQL error details to the user.
5. THE Platform SHALL allow users to save frequently used natural language queries to a personal query library.
6. WHEN a saved query is recalled, THE Platform SHALL re-execute it with the current `org_id` context.
7. THE NL_Query_Engine SHALL run LLM translation as an asynchronous Celery task on the `llm` queue.

### Requirement 16: Cross-Client Benchmarking

**User Story:** As a CA partner, I want to compare a client's financial ratios and compliance metrics against anonymized industry peers, so that I can provide evidence-based advisory recommendations.

#### Acceptance Criteria

1. THE Benchmarking_Engine SHALL be available only to Organizations on the premium subscription tier.
2. WHEN generating a benchmark comparison, THE Benchmarking_Engine SHALL require a minimum peer group of 5 clients from the same industry before presenting any comparison data.
3. THE Benchmarking_Engine SHALL anonymize all peer data and SHALL NOT expose the identity of any individual peer organization or client.
4. BEFORE including a client's data in any peer benchmark pool, THE Platform SHALL obtain explicit opt-in consent from the CA firm managing that client.
5. WHEN a firm has not opted in to benchmarking data sharing, THE Platform SHALL still allow that firm's clients to view benchmarking comparisons using peer data from opted-in firms.
6. THE Benchmarking_Engine SHALL compare metrics including GST compliance rate, ITC utilization percentage, average transaction volume, and health score distribution.

### Requirement 17: Frontend Dashboard and RBAC

**User Story:** As a CA firm partner, I want a responsive web dashboard where my team can manage clients, view reconciliations, and interact with AI features, with access controls matching each team member's role.

#### Acceptance Criteria

1. THE Platform SHALL provide a web frontend built with Next.js 14 that is accessible via a modern web browser.
2. THE Platform SHALL use AG Grid for rendering large transaction datasets with client-side column filtering and sorting.
3. WHEN a user logs in, THE Platform SHALL display only the navigation items and data permitted for their role (`partner`, `manager`, `article`).
4. THE Platform SHALL provide an Zoho Books connector that imports transactions and client data from Zoho Books into the Platform's data model.
5. THE Platform SHALL provide a Tally ERP connector that accepts CSV exports from both Tally Prime and Tally ERP 9 formats.
6. THE Platform SHALL render client health scores with color-coded visual indicators (red/amber/green) on the main client list.

### Requirement 18: Data Compliance and Security

**User Story:** As a CA firm partner, I want the platform to meet Indian data protection requirements and follow security best practices, so that I can use it for client data with confidence.

#### Acceptance Criteria

1. THE Platform SHALL store all data in AWS ap-south-1 (Mumbai) in compliance with the DPDP Act 2023 Indian data residency requirements.
2. THE Platform SHALL store all API keys, database credentials, and secret tokens exclusively in AWS Secrets Manager and SHALL NOT accept these values from environment variables committed to version control.
3. THE Platform SHALL enforce HTTPS on all public endpoints and SHALL reject unencrypted HTTP connections.
4. THE Platform SHALL enable automated RDS backups with a minimum 7-day retention period.
5. WHEN a user account is inactive for 30 days, THE Platform SHALL require re-authentication before granting access.
6. THE Platform SHALL log all authentication events, data access events, and API errors with timestamps and `org_id` for audit trail purposes.
7. THE Platform SHALL apply database row-level security such that even a misconfigured Scoped_Query cannot return data from a different Organization.
8. WHEN deploying to production, THE Platform SHALL run all services in Docker containers with no root-level process permissions.

### Requirement 19: Asynchronous Task Infrastructure

**User Story:** As a platform operator, I want all heavy processing tasks to run in background queues so that the API remains responsive and tasks can be retried on failure.

#### Acceptance Criteria

1. THE Platform SHALL use Celery with Redis as the message broker for all asynchronous task processing.
2. THE Platform SHALL route tasks to four dedicated queues: `whatsapp`, `ocr`, `heavy`, and `llm`.
3. WHEN a Celery task fails, THE Platform SHALL automatically retry the task with exponential backoff up to 3 attempts before marking it as failed.
4. THE Platform SHALL use Celery Beat to schedule the daily compliance check at 09:00 IST.
5. THE Platform SHALL expose a task status endpoint that allows the frontend to poll the status of long-running tasks (OCR, reconciliation, LLM generation) by task ID.

### Requirement 20: Document and Transaction Parsers

**User Story:** As a developer, I want all document parsers to produce consistent, validated output, so that downstream modules receive reliable structured data regardless of input format variation.

#### Acceptance Criteria

1. WHEN parsing a GSTR-2B JSON file, THE Platform SHALL validate it against the expected GSTR-2B schema and return a descriptive error if the schema does not match.
2. WHEN parsing a purchase register CSV, THE Platform SHALL validate that all required fields (`invoice_no`, `vendor_gstin`, `amount`, `date`) are present after normalization before inserting records.
3. THE Platform SHALL provide a pretty-printer that serializes Transaction objects back to a canonical CSV format.
4. FOR ALL valid Transaction objects, parsing the canonical CSV output with the Tally CSV normalizer SHALL produce an equivalent Transaction object (round-trip property).
5. WHEN a document parser encounters an unrecognizable format, THE Platform SHALL set the Document status to `parse_failed` with an error message and SHALL NOT insert partial transaction records.
